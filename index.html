what is going on in this repo · a code-aware RAG engine · TypeScript + Qdrant + local embeddings

Ask a codebase a question. Get a cited answer, or an honest refusal.

This repository is a RAG engine for source code (RAG: Retrieval-Augmented Generation, fetching the relevant passages from a corpus and handing them to a language model so it answers from evidence instead of memory). Point it at any repo and it indexes the code and docs; then you ask questions in plain English and get answers with path-and-line citations you can verify, or a refusal when the answer is not actually in the code. This page is the tour: how the pipeline works, why the design is what it is, what every setting does, and the part most demos skip, what happened when retrieval quality was actually measured. Short version: the first measurement was terrible, the eval diagnosed why, and one targeted change took the right-file rate from 11% to 61%.

📈 measured: right-file rate 11% → 61% 🧪 eval: free · offline · deterministic 📚 receipts: docs/DECISIONS.md · eval/results/

01 What this is, in one screen

Two flows sharing one database: build the index once, then answer every question against it.

The problem it solves

A language model answering from memory is a closed book

Ask a model cold, "how does this repo avoid corrupting files when it writes?", and it has never seen the repo. It will either say it does not know or, worse, produce a fluent, plausible, wrong answer. RAG turns the closed-book exam into an open-book one: before the model answers, the engine fetches the most relevant chunks of the actual code and says "answer only from these, and cite them."

Analogy: a brilliant new hire who has read the whole internet but has never seen your filing cabinet. Don't make them guess from memory: hand them the right folder before they answer. This engine is the folder-fetching.

One engine serves many repositories: each indexed repo gets its own isolated collection in the vector store, so one codebase never bleeds into another's answers.

The whole system on one screen. Top lane runs once per repository; bottom lane runs on every question. Qdrant, the vector database, is the shared hinge.

Every box is one small file

Each stage is an isolated module under src/, deliberately short, with the seams between them explicit. The entry points are thin on purpose: the CLI (cli/index.ts), the local web UI (src/web/), and the eval harness (src/eval/) are three front doors over the same pipeline functions, so adding a fourth (the planned one is an MCP server: Model Context Protocol, the standard plug that lets tools like Claude Desktop query the index directly) touches no pipeline code.

  • WALK → walk.ts: walks the repo, skipping build output and simple .gitignore patterns; ingest then classifies each file and keeps only known code/doc extensions (which excludes binaries for free).
  • CHUNK → chunkCode.ts (code, line windows) and chunk.ts (prose, character windows). Every chunk carries its path and line range, which is what makes path:line citations possible.
  • EMBED → embed.ts: a local model turns each chunk into a vector (a fixed-length list of numbers encoding its meaning). Runs offline, no API key. The model sits behind an Embedder interface, and that seam is the star of section 06.
  • STORE / RETRIEVE → store.ts / retrieve.ts: Qdrant holds the vectors; a question is embedded the same way and the nearest chunks by cosine similarity come back.
  • ANSWER → answer.ts: Claude answers strictly from the retrieved chunks, the response is schema-validated, and a refusal guardrail fires when retrieval is too weak.

02 The space the chunks live in

Every vector is scaled to length 1, so every chunk lives on the surface of one sphere. Meaning becomes direction. Spin it.

Interlude · what cosine similarity actually measures

Normalization turns meaning into direction

The embedder scales every vector to length 1 before it is stored. That has a picture worth holding: if every arrow out of the origin has the same length, then every chunk sits on the surface of the same unit sphere, and the only thing left to differ is which way each arrow points. Cosine similarity, the score Qdrant ranks by, is nothing more exotic than a readout of the from-the-center angle between two arrows (the score is the cosine of that angle): same direction means score near 1, unrelated means near 0.

Analogy: cities on a globe. Every city sits the same distance from the center of the Earth; what makes London and Paris "close" is the small angle between them as seen from the core. Chunks are cities, a question is one more city dropped onto the same globe, and retrieval is "list the nearest cities."

  • What to notice while you spin: everything huddles within about 52 degrees of the axis: the lowest top score on this run was 0.61 (acos of 0.61 is about 52 degrees), and no question, answerable or not, came near the 0.57 refusal floor. An unanswerable question still retrieves moderately similar real code, which is exactly why a score threshold alone cannot make the refusal call.
  • Cosine on the sphere gets even simpler: for length-1 vectors the cosine is the plain element-by-element multiply-and-sum of the two vectors (the dot product), no division needed. Qdrant's distance: Cosine setting means "rank by closeness on this globe."
  • The real space has 768 dimensions, one per number in each vector, where no picture fits. The sphere is a faithful 3-D shadow: these five stand-in points sit at the true angles their measured cosines dictate.

What the 11% turned out to mean

The first eval run found the expected file for only 2 of 18 questions. The diagnosis was manual and cheap: dump the top-8 retrieved chunks for every miss and read them. They were almost all markdown, READMEs and design docs describing the code, outranking the code itself. The first embedding model (all-MiniLM-L6-v2) is tuned on ordinary English prose, so an English question naturally lands nearest other English prose.

A probe made the diagnosis testable: restrict retrieval to code chunks only, re-run the eval, and hit-rate jumped to 61.1%. So the code was always reachable, just outranked. That number became a prediction: a code-tuned embedding model, trained on docstring-to-code pairs so English questions map near source code, should reach the same 61% with no restriction at all. It did (section 06). One wrinkle, visible in the per-item results: the probe and the swap agree at the rate level but share only 7 of their 11 hits; on an 18-question set, individual questions flip at thin margins.

04 The choices, and the roads not taken

Each of these is a fork recorded in docs/DECISIONS.md when it was taken: what was chosen, why, and what was considered and rejected.

FORK One generic tool with one collection per repo, not a repo per codebase. "Per-codebase" describes the data (a Qdrant collection), not the code. N copy-pasted RAG repos would drift apart and defeat the point of factoring out the common part. Rejected: rag-for-project-A, rag-for-project-B, the first instinct.
FORK TypeScript, reusing a proven document-RAG pipeline, not a rewrite. The pipeline was lifted from an earlier document-RAG project and extended with a code-aware front end. Rejected: a C#/.NET rewrite (discards working code for no retrieval benefit) and growing the original project in place (blurs its single-purpose story).
FORK Qdrant as the vector store. It was already running and fits the job: payload filtering, per-collection isolation, a local Docker container. Alternatives worth naming, not adopting: pgvector on Postgres, SQL Server 2025's native VECTOR type. Switching stores would be churn with no goal.
FORK Line-window chunking first, symbol-aware later, with the gain measured. Sixty-line windows with 15 lines of overlap are simple, language-agnostic, and give every citation a real path:line. The known weakness (a window can split a long function mid-body) is deliberate: it is the baseline that a tree-sitter upgrade must beat by a measured margin, not a guess.
FORK A free, local, code-tuned embedder over the paid cloud one the roadmap named. When the baseline demanded a code-tuned model, the candidates were voyage-code-3 (paid, cloud, needs a key) and jina-embeddings-v2-base-code (free, Apache-2.0, runs locally in the same pipeline). Cost had to be zero and the indexed repos are public, which removed the only reasons to prefer cloud. Free and local won, and section 06 shows it delivered.
FORK One model embeds both code and docs; a two-model design is parked. Cosine distances are only meaningful inside one model's vector space, so a single collection must use a single model. The code-tuned model handles English acceptably, and a dual-index, score-fusion design is more machinery than the gain justifies until prose-answered questions measurably regress.
FORK Grade retrieval, not answer prose, and never call the model in the eval. The question the eval answers is "does the right code come back," which is what chunking and embedding changes move. Keeping the model out makes a run free, offline, and deterministic: the same input gives the same number every time, which is what makes a baseline trustworthy. Rejected: a full-pipeline eval that grades generated prose (paid, nondeterministic, and no extra signal for the variable under test).

Every tunable lives in one file, src/config.ts, because each one is a tradeoff someone will eventually ask about.

The minScore story · a threshold cannot do this job

Why the refusal floor is a safety net, not a fake-question filter

The idea was simple: a question the codebase cannot answer should score low, so refuse below a floor. The measurement said otherwise. Answerable and unanswerable questions score in overlapping bands (on the current model: answerable 0.61 to 0.85, should-refuse probes 0.63 to 0.69), because an unanswerable question still retrieves moderately similar real code. You saw this on the globe in section 02: the pink dots sit inside the teal band.

  • The tuning was a measured tradeoff, not a dial-twiddle. On the first model, the floor was raised 0.25 → 0.57 after reading per-question scores: 0.57 caught 2 of the 4 probes at the cost of one wrongly-refused real question. A stricter 0.65 was tested and rejected: it caught all 4 probes but wrongly refused 6 of 18 real answers.
  • Then the model swap quietly retired the floor as a fake-filter. The code-tuned model lifted every score, and the probe band (0.63 to 0.69) settled entirely above the 0.57 floor. At the shipped configuration the threshold layer catches 0 of the 4 probes.
  • What refusal actually rests on now: the answer model reads the retrieved passages under an "answer only from context, else say you don't know" instruction, which is verified by spot checks but not yet scored by any eval. The floor remains as a safety net against near-empty retrieval. The measurable fix, a semantic judge that asks "do these passages actually answer the question?", is the top item in section 09.
Analogy: a metal detector at a door where every visitor, friendly or not, carries roughly the same amount of metal. However you set the sensitivity, you either wave everyone through or frisk everyone. The fix is not a better dial; it is a different question at the door.

06 The model switch, start to finish

The centerpiece change: symptom, diagnosis, prediction, a one-seam swap, and the measured result. Plus the bug it flushed out.

Text-tuned vs code-tuned

The old model kept ranking the description above the thing described

all-MiniLM-L6-v2 is a fine general-purpose embedder, trained on ordinary English. On a codebase that is precisely the problem: an English question lands nearest other English, so READMEs and design docs outrank the source they describe. jina-embeddings-v2-base-code is trained on roughly 150 million coding question-answer and docstring-to-code pairs across 30+ languages, so "how does X write big-endian integers?" maps near the C# that does it, not near the paragraph about it.

  • The swap was two config values, not a refactor. Every caller talks to an Embedder interface; no pipeline code knows which model runs. Changing localModelId and embeddingDim (384 → 768), then re-ingesting into a recreated collection, was the whole change. The seam was designed before it was needed, and this is the day it paid.
  • Why not the paid cloud model: voyage-code-3 was the roadmap's original pick. Against a zero budget and public target repos, a free, Apache-licensed, locally-running model with the same code-tuning premise removed every reason to pay or to ship code to a third party at index time.
  • Result, against the same golden set: hit-rate@8 11.1% → 61.1%, MRR 0.069 → 0.425. The gain landed exactly on the code-only probe's ceiling, which is what the diagnosis predicted. Ingest got slower (the model is roughly 7x larger: ~161M parameters vs MiniLM's ~22M), a real cost worth naming.
The bug the swap flushed out

A latent 32 MB wall, found by evidence, fixed at the source

Re-ingesting with the wider vectors died with a bare Bad Request. The real error was buried in the client error's data: the store sent all 2,588 points in one request, and 768-wide vectors pushed the body to ~44 MB against Qdrant's 32 MB request cap. The old 384-wide vectors had serialized to ~24 MB, just under the limit, so the bug had always been there and had simply never been hit. The fix went into the one shared function every ingest routes through: batch the upsert, 128 points per request, so any repo size and any vector width is safe. Both this and the diagnosis trail are written up in docs/PITFALLS.md.

07 How it is graded

"It demos fine" is not evidence. A hand-validated golden set and a model-free harness turn every tuning change into a measurable experiment.

The golden set and the three numbers

22 hand-validated questions, three scores, zero API calls

The golden set (eval/clipmeta.golden.json) pairs each question with the source file where the answer actually lives: 18 answerable questions plus 4 should-refuse probes, questions about things the target codebase genuinely does not do (transcoding, cloud upload, waveform rendering, a SQL database). The expected files were validated by hand against the real source, because grading a tool against its own output is circular and would score high while meaning nothing.

  • Hit-rate@8 (hit@8 for short): the share of answerable questions whose expected file lands anywhere in the top 8 retrieved chunks. The headline number, because if the right file never reaches the model, no generation can recover.
  • MRR (Mean Reciprocal Rank): each question scores 1 divided by the position of the first matching chunk (a miss scores 0), averaged across questions. It rewards ranking the answer first over sneaking it in eighth, resolution that a pass/fail hit-rate throws away.
  • Refusal-accuracy: the share of the 4 probes whose top score falls below the refusal floor, mirroring the exact rule the live guardrail uses. At the shipped configuration this is 0 of 4, and section 05 explains why that number is reported instead of hidden.
  • Model-free on purpose: both checks are computed from retrieval output plus the threshold. A run costs nothing, works offline, and gives the same number every time, which is what makes before/after comparisons trustworthy.

One more detail the per-item results add: the highest-scoring miss (0.796) outscored most hits. A cosine score is confidence in similarity, not correctness; the eval exists precisely because scores alone cannot tell you retrieval is working.

Known scope limits: every answerable question targets a code file, so "the code-tuned model still embeds the docs acceptably" is an assumption the current set cannot check; and answer prose quality is not graded at all, because a model-graded eval is deliberately outside the free deterministic harness. Both are on the roadmap.

08 Running it

Node, Docker, and an Anthropic API key in .env. Indexing runs fully offline; only the answer step calls an API.

The commands
docker compose up -d                                  # start Qdrant (the vector store)
npm install
npm test                                              # unit tests (also run in CI on every push)

npm run ingest -- --repo myproject --dir ../myproject # walk → chunk → embed → store
npm run ask   -- --repo myproject "how does X work?"  # cited answer or refusal
npm run eval  -- --repo clipmeta                      # score retrieval vs the golden set
npm run web                                           # local web UI at 127.0.0.1:5175

The web UI is a second front door over the same pipeline: a repository picker, answers with clickable citations that expand to the retrieved code, a retrieval panel that plots each chunk's score against the refusal floor, and two live-tunable knobs (topK, minScore) saved per repository.

▶ Where the code goes, stage by stage

Ingest, embedding, storage, and retrieval run entirely on the local machine: the embedding model is downloaded once and runs in-process, and Qdrant is a local container. The eval never leaves the machine either. The answer step is the exception: the retrieved chunks and the question are sent to the Anthropic API so Claude can write the cited answer. The API key lives server-side in .env and is never shipped to a browser.

09 Sharp edges and what comes next

Known limits are documented in docs/PITFALLS.md as they are found. The roadmap items below are each backed by a measured need, not a vibe.

Known limits, today
  • Refusal of unanswerable questions rests on the answer model, not the threshold (section 05), and that layer is spot-checked, not yet eval-scored.
  • Line-window chunking can split a long function mid-body. Deliberate baseline; the symbol-aware upgrade must prove itself against the 61% number.
  • .gitignore support is intentionally partial (simple patterns from the repo's top-level file only; no negation, no nested .gitignore files). Binaries are excluded anyway by the extension allow-list.
  • Re-ingesting upserts and does not delete stale chunks. Renamed or deleted files leave orphans; the clean path today is dropping the collection and rebuilding.
Next, in order of measured need
  • A semantic refusal judge: score "do these passages actually answer the question?" so refusal becomes measurable instead of threshold-guessed. The 0-of-4 number is the motivation.
  • Tree-sitter symbol-aware chunking: chunks on function/class boundaries carrying the symbol name, measured against the line-window baseline.
  • An MCP server (Model Context Protocol, the standard plug that lets tools like Claude Desktop query an index directly): a fourth thin front door over the same pipeline.
  • Parked with reasons (in DECISIONS.md): hybrid keyword+vector search, re-ranking, GraphRAG, hosted embeddings. Each waits for a measured need.
